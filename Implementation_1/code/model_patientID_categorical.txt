# Build supervised rows
# Each row becomes: predictors from the past -> current pre-meal glucose target.

def build_modeling_rows(patient_df):
    rows = []
    history = []

    for _, row in patient_df.iterrows():
        code = row["Code"]
        t = row["AbsMinute"]
        value = row["ValueNum"]

        # If this row is a target glucose measurement, create one training example.
        if code in TARGET_CODES and pd.notna(value):
            prior = [event for event in history if event["time"] < t]

            def prior_events(codes, hours):
                cutoff = t - hours * 60
                return [e for e in prior if e["code"] in codes and e["time"] >= cutoff]

            def sum_values(codes, hours):
                events = prior_events(codes, hours)
                return sum(e["value"] for e in events if pd.notna(e["value"]))

            def count_events(codes, hours):
                return len(prior_events(codes, hours))

            previous_glucose = [
                e for e in prior
                if e["code"] in GLUCOSE_CODES and pd.notna(e["value"])
            ]

            if previous_glucose:
                last_glucose = previous_glucose[-1]
                prev_glucose_value = last_glucose["value"]
                hours_since_prev_glucose = (t - last_glucose["time"]) / 60
            else:
                prev_glucose_value = np.nan
                hours_since_prev_glucose = np.nan

            minute = row["MinuteOfDay"]

            rows.append({
                "patientId": row["patientId"],
                "target_code": code,
                "Day": row["Day"],
                "MinuteOfDay": minute,
                "time_sin": np.sin(2 * np.pi * minute / 1440),
                "time_cos": np.cos(2 * np.pi * minute / 1440),

                "prev_glucose": prev_glucose_value,
                "hours_since_prev_glucose": hours_since_prev_glucose,

                "regular_insulin_8h": sum_values({"33"}, 8),
                "nph_insulin_24h": sum_values({"34"}, 24),
                "ultralente_insulin_24h": sum_values({"35"}, 24),

                "typical_meal_6h": count_events({"66"}, 6),
                "more_meal_6h": count_events({"67"}, 6),
                "less_meal_6h": count_events({"68"}, 6),

                "typical_exercise_6h": count_events({"69"}, 6),
                "more_exercise_6h": count_events({"70"}, 6),
                "less_exercise_6h": count_events({"71"}, 6),

                "hypo_symptoms_24h": count_events({"65"}, 24),
                "special_event_24h": count_events({"72"}, 24),

                "glucose": value,
            })

        # Add current event after feature creation so same-time insulin is not leaked.
        history.append({
            "time": t,
            "code": code,
            "value": value,
        })

    return rows

# Custom objective function for the assignment writeup
# This matches the elastic net objective:
# MSE term + L1 sparsity penalty + L2 stability penalty.

def elastic_net_objective(y_true, y_pred, beta, lambda_value, l1_ratio):
    mse_term = 0.5 * np.mean((y_true - y_pred) ** 2)
    l1_term = lambda_value * l1_ratio * np.sum(np.abs(beta))
    l2_term = 0.5 * lambda_value * (1 - l1_ratio) * np.sum(beta ** 2)
    return mse_term + l1_term + l2_term

Answer: 


What The Row Builder Does
That function converts the event log into a supervised learning table.

The raw data is like this:

patient event at time t
patient event at time t
patient event at time t
The model needs this instead:

past information before time t -> current glucose value at time t
So every time the loop finds a target glucose code 58, 60, 62, 64, it creates one model row. The target is the current glucose value, and the predictors are built from earlier rows only.
__________
time_sin and time_cos represent time of day cyclically. MinuteOfDay = 1439 and MinuteOfDay = 0 are far apart numerically, but in real life they are only one minute apart. These features encode the daily cycle:
time_sin = np.sin(2 * np.pi * minute / 1440)
time_cos = np.cos(2 * np.pi * minute / 1440)
__________
Ridge, LASSO, Elastic Net
These are all regression models. They fit a weighted linear equation:
predicted glucose = b0 + b1*x1 + b2*x2 + ... + bp*xp
So yes, conceptually it is a best-fit line, but in many dimensions.

Ridge: keeps all predictors but shrinks coefficients. Good for correlated features.
LASSO: can force some coefficients exactly to zero. Good for feature selection.
Elastic Net: mix of Ridge and LASSO. It can select predictors while being more stable when features are correlated.

This part:
("imputer", SimpleImputer(strategy="median")),
("scaler", StandardScaler()),
("model", model),

means:
SimpleImputer: fills missing values. Median means missing prev_glucose or hours_since_prev_glucose becomes the median training value. Other options include "mean", "most_frequent", or a constant.
StandardScaler: normalizes numeric columns to mean 0 and standard deviation 1. This is important because insulin doses, glucose values, day numbers, and one-hot patient IDs are on different scales.
model: the actual Ridge, LASSO, or Elastic Net model.

Scaling does not prove causality, but it prevents the model from treating larger-scale variables as automatically more important.
_______________________________
Metrics
Your table says:
MSE: average squared error
RMSE: square root of MSE, in glucose units
MAE: average absolute error, in glucose units
Example: RMSE = 70 means predictions are often off by about 70 mg/dL, with larger errors punished more strongly. MAE = 55 means the average absolute miss is about 55 mg/dL.

MSE is harder to interpret directly because the unit is squared glucose.
For medical context, common diabetes glucose targets are roughly 80-130 mg/dL before meals and under 180 mg/dL about 1-2 hours after meals, while below 70 mg/dL is generally considered low. Sources: CDC and NIDDK.
_______________________________
Coefficient Plot
The coefficient plot shows which predictors Elastic Net selected and how strongly they move the prediction.

Positive coefficient:

higher feature value -> higher predicted glucose
Negative coefficient:

higher feature value -> lower predicted glucose
A bigger absolute value means stronger influence after scaling. Lower is not “better.” Zero means the model did not use that feature.
_______________________________
Custom Objective Function
This is not another model. It is just calculating the Elastic Net objective value after the model is already trained:

objective = MSE term + L1 sparsity penalty + L2 stability penalty
alpha is the regularization strength chosen by cross-validation. Higher alpha means stronger penalty.

l1_ratio controls the LASSO-vs-Ridge mix:
l1_ratio = 1.0 -> pure LASSO
l1_ratio = 0.0 -> pure Ridge-like penalty
l1_ratio = 0.95 -> mostly LASSO with a little Ridge
The objective value is useful for your optimization writeup, but do not compare it directly to RMSE/MSE baselines. It includes penalty terms, so it is not the same performance metric.
